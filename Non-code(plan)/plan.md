## 1. Product Vision

**VibeDoc** takes a directory of source code and outputs a single Markdown (or PDF) document that serves as a **comprehensive, trustworthy manual**. It doesn’t just restate syntax; it explains:

- **Architecture & purpose** of the system.
- **Static structure** — every module, function, class, dependency.
- **Dynamic behaviour** — actual execution flows, data transformations, side effects.
- **Semantic domain model** — business entities, rules, state machines, configuration.
- **Code health warnings** — dead code, anti‑patterns, missing error handling.

The tool targets codebases generated by AI (“vibe coded”) where no human truly understands the full system. It must work with zero user input, auto‑generating scenarios and synthesising insights.

## 2. Core Architecture

The pipeline comprises five phases. The first three are always executed; Phase 4 and 5 are optional but strongly recommended for deep understanding.

```
[Repo]
   │
   ├─(1) STATIC PARSE & GRAPH ──► AST + Dependency Graph
   │
   ├─(2) STATIC EXPLANATION ────► Per‑component analyses (LLM)
   │
   ├─(3) RUNTIME AWARENESS ─────► Trace logs / simulated trace
   │
   ├─(4) SEMANTIC SYNTHESIS ────► Domain model, flows, config map
   │
   └─(5) DOCUMENT ASSEMBLY ─────► Final Markdown → PDF
```

Each phase is a pipeline step with its own LLM calls and intermediate data structures.

## 3. Detailed Plan

### The Milestone‑Based Master Plan

### Milestone 0 – Project Skeleton & LLM Integration
**Goal:** Have a runnable CLI that can take a code file and talk to an LLM.

**Deliverables:**
- CLI with `click` or `typer` (`vibedoc analyze <file>`)
- LLM client abstraction (Ollama, OpenAI, or Anthropic, LMstudio)
- Configuration via environment variables or `.env`
- Basic logging

**Key tasks:**
- Initialize Python project with `pyproject.toml`
- Set up CLI entry point that accepts `--file`, `--model`, `--output`
- Create `llm_client.py` with `def query(system_prompt, user_prompt) -> str`
- Test by sending a hardcoded prompt and printing the response

---

### Milestone 1 – Single‑File Static Analyzer
**Goal:** Deeply explain one source file (all functions/classes) using only static analysis.

**Deliverables:**
- AST parser for one language (Python via built‑in `ast` or tree‑sitter)
- Dependency graph for that file (calls between functions, imports)
- Context injection (imports, callee signatures)
- Function‑level explanations using the specialised system prompt
- Single Markdown output with function details

**Key tasks:**
- Parse file, extract function/class definitions and imports
- Resolve internal function calls and build a mini call graph
- Create chunks: for each function, prepend imports and signatures of called functions
- Write the **function explanation prompt** (the one in Phase 2 of the full plan)
- Call LLM per function, store results
- Generate a simple Markdown file with the explanations and a table of contents

**Test:** Run on a messy, vibe‑coded Python file. The output should read like a senior dev’s analysis, not a syntax rephrase.

---

### Milestone 2 – Multi‑File Repository & Module Synthesis
**Goal:** Handle an entire directory, producing module‑level guides and a unified static manual.

**Deliverables:**
- Recursive directory scanning
- Full project dependency graph (imports, inter‑file calls)
- Bottom‑up processing order (leaves first)
- Module‑level synthesis (combining all function docs of a file)
- Assembled static manual (Markdown) with section per module
- Simple caching (.vibedoc_cache) to avoid re‑explaining unchanged code

**Key tasks:**
- Extend parser to walk a directory, filter by extension
- Build a `networkx` graph: nodes are files and functions; edges are `imports` and `calls`
- Topological sort to process leaves first; handle cycles gracefully
- For each module, after all its functions are explained, call the **module synthesis prompt**
- Use Jinja2 to stitch all module docs into a single `report.md`
- Implement cache keyed on `(source_hash, prompt_version)`

**Test:** Document a small project (e.g., a Flask API with 5–10 files). The manual should have an overview, API details, and a quickstart snippet.

---

### Milestone 3 – Runtime Awareness via LLM Simulation
**Goal:** Add a `--runtime simulate` mode that infers execution traces without running the code.

**Deliverables:**
- Scenario generation: ask LLM to propose key user stories based on static docs
- Simulated traces for each scenario (function call sequence, data flow)
- Mermaid sequence diagrams for the traces
- New “Dynamic Behaviour” chapter in the final document

**Key tasks:**
- Create a **scenario generator prompt**: “Based on this system documentation, list 5 typical user scenarios with step‑by‑step descriptions.”
- For each scenario, feed the static documentation (module guides, function summaries) into the **runtime simulator prompt**: “Simulate a detailed execution trace for the scenario: …”
- Parse the LLM output into a structured trace (JSON) and extract a Mermaid sequence diagram
- Add a “Dynamic Behaviour & Scenarios” section to the Jinja2 template, populating it with scenarios and diagrams

**Test:** On a codebase you understand well, run `vibedoc analyze ./repo --runtime simulate`. Check if the simulated traces match the actual logic; they should capture conditional branches and hidden flows.

---

### Milestone 4 – Semantic Synthesis & Full Document
**Goal:** Produce a domain model, architecture overview, configuration map, and code health report—all by reasoning over static + simulated dynamic data.

**Deliverables:**
- Domain entity extraction (entities, attributes, relationships, state machines)
- Architecture synthesis (C4 container diagram, design patterns, anti‑patterns)
- Configuration & environment map
- Code health warnings (dead code, missing error handling, inconsistent naming)
- Final polished manual with all sections, hyperlinks, and table of contents
- PDF output via `pandoc` or `weasyprint`

**Key tasks:**
- Write and tune the **domain entity extraction prompt**, **architecture synthesis prompt**, and **configuration map prompt** (from Phase 4)
- Feed all accumulated knowledge (static function/module docs + simulated traces + dependency graph) into these prompts
- Create a comprehensive Jinja2 template that includes:
  - System Architecture (with Mermaid diagrams)
  - Domain Model
  - Dynamic Behaviour & Scenarios
  - Module Reference
  - Configuration & Environment
  - Code Health & Warnings
- Implement `--format pdf` using a subprocess call to `pandoc`
- Add cross‑references (e.g., link from a function to its callers/callees using Markdown anchors)

**Test:** The final manual of a modest project should be such that a new developer can understand the entire system, deploy it, and start modifying it safely.

---

### Milestone 5 – Instrumented Execution (Real Traces)
**Goal:** Add `--runtime instrument` to actually run the code and capture real traces, replacing simulated traces with ground truth.

**Deliverables:**
- LLM‑generated driver script that exercises the system’s main entry points
- AST‑based automatic decorator injection that logs every function call with args/return values/exceptions
- Sandboxed execution in a temporary environment
- Trace log parser and integrator into the existing semantic pipeline
- Comparison mode that highlights differences between simulated and real traces

**Key tasks:**
- Write a **driver generator prompt**: “Write a Python script that imports and exercises the main flows of this app; use realistic sample data.”
- Build an AST transformer that wraps every function definition with a `@trace` decorator (or equivalent in other languages)
- Implement a sandbox runner that sets up a clean virtualenv, installs requirements, runs the driver, and captures logs
- Parse the resulting JSON‑lines log into the same trace format used in Milestone 3
- Feed real traces (instead of simulated) into the dynamic behaviour and semantic synthesis prompts
- Optionally, add a “Trace Fidelity” report showing where simulation was wrong

**Test:** On a runnable vibe‑coded app, the real traces should expose dynamic imports, lazy‑loaded modules, and runtime configuration effects that static+simulation missed.

---

### Milestone 6 – Multi‑Language & Advanced Features
**Goal:** Make the tool robust and usable on a variety of codebases, with QoL improvements.

**Deliverables:**
- Support for JavaScript/TypeScript via tree‑sitter grammars
- `--lang auto` detection
- Incremental mode: re‑analyze only changed files using the cache
- Interactive output (e.g., an HTML manual with searchable, collapsible sections)
- Plugin system for custom language support or custom analysis passes
- Distributable as a PyPI package or single binary (via `pex` or `shiv`)

**Key tasks:**
- Abstract the parsing layer behind a language‑specific interface; implement for Python and JS/TS
- Use file extensions and heuristics to auto‑detect language
- Enhance caching to track dependency changes; invalidate only affected chunks
- Generate HTML with `pandoc` or a Jinja2 HTML template
- Write a `setup.py` / `pyproject.toml` for PyPI publishing
- Add `--verbose` and `--quiet` flags for user control

**Test:** Run against a mixed Python+JS repository (e.g., a FastAPI backend with a Svelte frontend). The manual should seamlessly cover both parts.

## Milestone Dependencies & Flow
 
- **M1–M2** are the static foundation. Everything else builds on them.
- **M3 (simulated runtime)** can start once M2’s module docs are solid, because the simulator uses them as input.
- **M4 (semantic synthesis)** needs both static docs and traces (from M3 or M5) to produce the full manual.
- **M5 (instrumented)** is optional but high‑value; it replaces the simulated traces with real ones. You can first ship with M3, then add M5 later.
- **M6** makes the tool production‑ready and broadens its reach.

## How to Approach Each Milestone

1. **Define success criteria**—what a “done” milestone looks like (I gave examples above).
2. **Build the minimal thing** that achieves the deliverable, test it manually on a tiny vibe‑coded project.
3. **Tweak prompts** relentlessly; the system prompts and their context are the soul of the tool.
4. **Commit the intermediate output** (e.g., the Markdown manual) to see progress; it’s deeply satisfying.
5. **When stuck**, fall back to the detailed phase descriptions I provided earlier—they contain the exact prompts, data structures, and logic for each step.

This milestone plan keeps you focused on shipping incrementally while gradually injecting the runtime awareness and semantic understanding that will make VibeDoc unique. Start at M0, and within a few milestones you’ll have a tool that genuinely explains what a vibe‑coded mess *really* does. Good luck!


## 4. CLI Design

```
vibedoc analyze <target_directory> [OPTIONS]

Options:
  --lang <python|javascript|typescript>   (auto‑detect if omitted)
  --output <file>                         (default: ./VIBEDOC_MANUAL.md)
  --format <md|pdf>                       (default: md)
  --runtime <off|simulate|instrument>     (default: simulate)
  --model <openai|ollama|claude|LMstudio>          (LLM backend)
  --api-key <key>                         
  --max-depth <n>                         (max call depth for graph, default 5)
  --include-tests                         (also document test files)
  --cache-dir <path>                      (default: .vibedoc_cache)
```

**Example:**
```bash
vibedoc analyze ./my_vibe_app --runtime simulate --model ollama --output docs/MANUAL.md
```

## 6. Key Design Decisions & Pitfalls to Avoid

- **Stay local‑first:** Use a free local model (e.g., `llama3.1‑8b` or `deepseek-coder-v2` via Ollama) for unlimited iterations. Only use paid APIs for final polish if needed.
- **Cache aggressively:** LLM calls are the bottleneck. Base cache keys on `(source_code_hash, prompt_version_hash)`.
- **Handle huge repos:** stream chunks and write the document progressively; don’t hold everything in memory.
- **Security:** If you add real execution, run it in a Docker container or a temporary venv with limited permissions.
- **Trust but verify:** The LLM will hallucinate. Always link explanations back to source lines (you can add source annotations like `(file.py:12‑34)`). Encourage the user to treat the document as a “starting point for review”.
